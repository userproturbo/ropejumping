# ropejumping

## 📄 Концепция проекта

**ropejumping** — веб-платформа для роупджампинг-сообщества, которая объединяет команды, участников, мероприятия, объекты, историю участия и медиа-контент.

Главная идея проекта — создать не просто сайт с анонсами прыжков, а живую систему, где человек может найти мероприятие, подать заявку, попасть на прыжок, сохранить свой опыт в профиле и стать частью сообщества.

---

# 1. 🧠 Идея проекта

Платформа объединяет:

* команды по роупджампингу;
* участников: новичков и опытных;
* мероприятия;
* объекты;
* фото и видео;
* историю участия;
* бейджи и достижения;
* безопасное информационное пространство вокруг роупджампинга.

## Цель

Сделать единое место, где можно:

* находить прыжки;
* организовывать мероприятия;
* подавать заявки на участие;
* общаться с командами и участниками;
* делиться фото, видео и историями;
* сохранять личную историю участия;
* видеть опыт пользователей и команд;
* безопасно вводить новичков в тему.

# 3. 👥 Пользовательские роли

## 3.1 Пользователь

Базовая роль.

Пользователь может:

* зарегистрироваться;
* создать профиль;
* подавать заявки на мероприятия;
* писать посты;
* комментировать;
* загружать фото и видео;
* получать бейджи;
* состоять в командах;
* видеть свою историю участия.

## 3.2 Команда

Команда — это публичная страница группы организаторов или сообщества.

Команда может:

* иметь страницу с описанием;
* публиковать мероприятия;
* публиковать посты;
* добавлять участников;
* показывать историю проведённых событий;
* получать статус «проверенная команда».

## 3.3 Роли внутри команды

Для MVP достаточно простой модели:

* **Владелец** — управляет командой, ролями и настройками;
* **Администратор** — редактирует страницу команды, участников и контент;
* **Организатор** — создаёт мероприятия и управляет заявками;
* **Участник команды** — отображается в составе команды, но не обязательно имеет права управления.

Дополнительные роли вроде «фотограф», «инструктор», «старший организатор» лучше делать не как права доступа, а как бейджи или отображаемые статусы.

## 3.4 Организатор

Организатор — роль внутри команды.

Организатор может:

* создавать мероприятия;
* редактировать мероприятия;
* принимать и отклонять заявки;
* закрывать набор;
* отмечать мероприятие как состоявшееся;
* подтверждать фактических участников после мероприятия.

## 3.5 Модератор / админ платформы

Админ может:

* проверять команды;
* выдавать статус «проверенная команда»;
* рассматривать жалобы;
* скрывать и удалять контент;
* блокировать пользователей;
* скрывать опасные объекты или мероприятия;
* следить за тем, чтобы на платформе не публиковались опасные инструкции.

## 5.2 Лента

Лента — это живая хроника роупджампинга.

Она нужна, чтобы:

* показать, что на платформе есть жизнь;
* объединять людей;
* продвигать мероприятия и команды;
* давать место для фото, видео, историй и обсуждений.

### Принцип ленты

Лента должна быть:

* простой;
* визуальной;
* понятной;
* без сложных алгоритмов на старте.

На MVP не нужна «умная лента». Достаточно хронологической ленты с базовыми фильтрами.

### Типы постов

#### 1. Анонс мероприятия

Создаётся автоматически, когда организатор публикует мероприятие.

Содержит:

* фото;
* название;
* дату;
* объект;
* команду;
* кнопку «Перейти к мероприятию».

#### 2. Отчёт с мероприятия

Создаётся после события.

Может содержать:

* фото;
* видео;
* текст;
* отметку мероприятия;
* отметку объекта;
* отметку команды.

#### 3. Медиа-пост

Обычный пост с фото или видео.

Можно привязать к:

* объекту;
* мероприятию;
* команде.

#### 4. Текстовый пост

Для историй, вопросов, мыслей и обсуждений.

#### 5. Вопрос

Отдельный тип поста можно добавить позже, чтобы стимулировать общение.

#### 6. Системные события

Например:

* пользователь получил бейдж;
* команда провела мероприятие;
* добавлен новый объект.

Важно: системные события не должны превращаться в спам.

### Структура поста

Верх:

* аватар;
* имя автора;
* тип автора: пользователь или команда;
* дата;
* привязка к мероприятию, объекту или команде.

Контент:

* текст;
* фото или видео;
* теги.

Низ:

* лайк;
* комментарий;
* меню: пожаловаться, скрыть.

Репосты лучше не добавлять в MVP.

### MVP ленты

Для первой версии достаточно:

* постов;
* фото;
* лайков;
* комментариев;
* автоматических анонсов мероприятий;
* привязки поста к объекту и мероприятию;
* кнопки жалобы.

## 5.3 Мероприятия

Мероприятия — ключевой раздел платформы.

Это основа всей системы, потому что через мероприятия формируются:

* заявки;
* участие;
* история пользователя;
* статистика;
* бейджи;
* активность команды;
* история объекта.

### Карточка мероприятия

Карточка содержит:

* название;
* фото;
* команду;
* объект;
* дату;
* регион;
* высоту;
* количество мест;
* количество свободных мест;
* уровень;
* статус;
* короткое описание;
* кнопку действия.

### Статусы мероприятия

Лучше сразу использовать понятный жизненный цикл:

* **Черновик** — видит только организатор;
* **Опубликовано** — мероприятие видно пользователям;
* **Набор открыт** — можно подавать заявки;
* **Мест нет** — лимит принятных участников достигнут;
* **Набор закрыт** — заявки больше не принимаются;
* **Перенесено** — дата изменилась;
* **Отменено** — событие не состоится;
* **Состоялось** — мероприятие прошло;
* **Архив** — событие завершено и хранится как история.

### Главный сценарий

1. Организатор создаёт мероприятие.
2. Мероприятие публикуется.
3. Пользователи подают заявки.
4. Организатор принимает или отклоняет заявки.
5. Мероприятие проводится.
6. Организатор нажимает «Мероприятие состоялось».
7. Система показывает список принятых участников.
8. Организатор отмечает, кто фактически участвовал.
9. Система фиксирует участие.
10. Пользователям начисляются бейджи.
11. Мероприятие становится частью истории пользователей, команды и объекта.

### Важное изменение

Нельзя автоматически засчитывать участие всем, кто был принят на мероприятие.

Причины:

* человек мог не приехать;
* мог приехать, но не прыгать;
* мог быть зрителем;
* мог отказаться на месте;
* мог быть добавлен в список по ошибке.

Поэтому после события нужен простой экран подтверждения участников.

Организатор должен сделать минимальное действие:

> «Отметить, кто действительно участвовал»

Это сохраняет принцип минимальной нагрузки, но делает статистику честной.

## 5.4 Страница мероприятия

Страница мероприятия — ключевой экран проекта.

Она отвечает на вопросы:

* что это за событие;
* кто организует;
* где примерно проходит;
* какой уровень нужен;
* можно ли подать заявку;
* кто уже принят;
* что произошло после события.

### Структура страницы

#### 1. Hero-блок

Содержит:

* большое фото или видео;
* название;
* дату;
* регион;
* команду;
* статус;
* кнопку действия.

Пример:

**Прыжки с моста «Северный»**
18 мая 2026 · Московская область · Rope Team
Набор открыт · 6 мест осталось

Кнопка:

* Подать заявку;
* Заявка отправлена;
* Вы приняты;
* Набор закрыт.

#### 2. Быстрые факты

* высота;
* тип объекта;
* количество мест;
* свободные места;
* уровень;
* цена, если есть;
* возрастные ограничения, если есть.

#### 3. Требования

Отдельный заметный блок.

Пример:

* опыт: желательно 20+ прыжков;
* высота: желательно 70+ м;
* новичкам: нет;
* возраст: 18+;
* физическое состояние: без противопоказаний.

Под блоком:

> Решение о допуске принимает организатор. Система не блокирует заявки автоматически.

#### 4. Описание

Организатор описывает:

* как проходит день;
* время сбора;
* примерный формат;
* что взять с собой;
* ограничения;
* важные условия;
* общие особенности события.

Важно не публиковать опасные технические детали.

#### 5. Объект

Мини-карточка объекта:

* название;
* фото;
* тип;
* высота;
* регион;
* ссылка на объект.

#### 6. Участники

Показывать:

* количество принятых;
* аватарки принятых участников;
* список участников, если это разрешено настройками приватности.

#### 7. Заявки для организатора

Организатор видит:

* новые заявки;
* принятые заявки;
* отклонённые заявки;
* комментарии пользователей;
* опыт пользователя на платформе;
* ключевые бейджи;
* кнопки «Принять» и «Отклонить».

#### 8. Обсуждение

Комментарии под мероприятием:

* вопросы;
* уточнения;
* логистика;
* общение участников.

#### 9. После мероприятия

После статуса «Состоялось» страница превращается в архив:

* список подтверждённых участников;
* фото и видео;
* отчёт команды;
* связанные посты;
* выданные бейджи;
* история объекта.

## 5.5 Создание мероприятия

Главный принцип:

> Создание мероприятия должно быть похоже на создание поста, но с несколькими структурными полями.

Если форма будет сложной, организаторы не будут ей пользоваться.

### Структура формы

#### 1. Основная информация

Обязательные поля:

* название;
* дата;
* команда;
* объект;
* количество мест.

Опциональные поля:

* регион;
* цена;
* уровень;
* обложка;
* описание;
* требования.

#### 2. Объект

Два варианта:

1. Выбрать существующий объект.
2. Создать новый объект в упрощённой форме.

При создании объекта указываются только безопасные данные:

* название;
* тип;
* высота;
* регион;
* фото;
* общее описание.

Не указываются:

* точные координаты;
* схемы;
* точки доступа;
* технические детали;
* способы организации прыжка.

#### 3. Быстрые параметры

* высота — автоматически из объекта;
* количество мест;
* цена, если есть;
* уровень: новичкам можно / с опытом / только опытные.

#### 4. Требования

Простые текстовые поля:

* опыт;
* желательная высота;
* возраст;
* ограничения;
* дополнительные условия.

Можно добавить кнопку «Заполнить шаблон».

#### 5. Описание

Большое текстовое поле с подсказками:

* как проходит день;
* что взять;
* примерный тайминг;
* условия;
* важные нюансы.

#### 6. Обложка

Можно:

* загрузить фото;
* выбрать фото объекта;
* использовать обложку команды.

#### 7. Публикация

Кнопки:

* Опубликовать;
* Сохранить как черновик.

После публикации система автоматически:

* создаёт страницу мероприятия;
* создаёт анонс в ленте;
* открывает заявки;
* отправляет уведомления подписчикам команды, если такая механика включена.

# 6. 📩 Заявки

Заявки — это место, где организатор принимает решение.

Главная задача:

* быстро понять, кто подал заявку;
* быстро принять или отклонить;
* не заставлять организатора открывать 10 разных страниц.

## Статусы заявки

* **Ожидает** — заявка подана;
* **Принята** — организатор принял пользователя;
* **Отклонена** — организатор отказал;
* **Отменена пользователем** — пользователь сам отменил заявку;
* **Был на месте** — можно добавить позже;
* **Участие подтверждено** — пользователь реально участвовал;
* **Не явился** — пользователь был принят, но не участвовал.

Для MVP достаточно:

* ожидает;
* принята;
* отклонена;
* отменена пользователем;
* участие подтверждено;
* не явился.

## Экран заявок для организатора

Лучше сделать как вкладку внутри мероприятия.

Верхний блок:

* название мероприятия;
* дата;
* количество мест;
* принято / всего;
* количество заявок.

Фильтры:

* все;
* новые;
* принятые;
* отклонённые;
* отменённые.

## Карточка заявки

Карточка содержит:

* аватар;
* имя;
* город;
* комментарий к заявке;
* опыт по данным платформы;
* опыт по словам пользователя;
* ключевые бейджи;
* дату подачи заявки;
* кнопки «Принять» и «Отклонить».

Пример опыта:

* 5 мероприятий на платформе;
* 3 объекта;
* максимальная высота 80 м;
* опыт по словам пользователя: «прыгаю 3 года».

## Важная логика

Система не должна автоматически запрещать заявки.

Она только показывает информацию:

* историю пользователя;
* бейджи;
* комментарий;
* опыт на платформе;
* опыт по словам пользователя.

Решение всегда принимает организатор.

## Когда мест не осталось

Если лимит принятных участников достигнут, можно:

* автоматически показать статус «Мест нет»;
* скрыть кнопку подачи заявки;
* или разрешить заявки в резервный список.

Для MVP лучше сделать простой вариант:

* при достижении лимита статус меняется на «Мест нет»;
* организатор всё равно может вручную увеличить лимит или принять человека сверх лимита.

---

# 7. 👤 Профиль пользователя

Профиль — это история человека в роупджампинге.

Он должен отвечать на вопрос:

> Кто этот человек в роупджампинге?

## Главные задачи профиля

* показать опыт человека;
* показать участие в сообществе;
* помочь организатору понять уровень;
* сохранить историю участия;
* мотивировать человека возвращаться на платформу.

## Структура профиля

### 1. Верхний блок

Содержит:

* аватар;
* имя или ник;
* город;
* короткое описание;
* 2–4 ключевых бейджа;
* кнопку «Подписаться» — можно добавить позже.

### 2. Опыт

Показывать:

* количество подтверждённых мероприятий;
* количество уникальных объектов;
* максимальную высоту;
* команды, в которых состоит пользователь.

Важно: на MVP не считать количество прыжков вручную.

### 3. Опыт по словам пользователя

Нужно отдельное поле:

> Опыт вне платформы

Пример:

«Прыгаю с 2017 года, был на мостах 40–120 м».

Это важно, потому что опытные люди могут зарегистрироваться на сайте уже после многих лет практики.

Такой опыт не должен автоматически превращаться в бейджи, но должен помогать организатору принимать решение.

### 4. Бейджи

Категории:

* участие;
* объекты;
* высота;
* активность;
* роли;
* особые бейджи.

### 5. Объекты

Показывать объекты, где пользователь участвовал в подтверждённых мероприятиях:

* фото;
* название;
* высота;
* дата первого участия.

### 6. История

Таймлайн:

* участие в мероприятиях;
* получение бейджей;
* публикации;
* фото и видео.

### 7. Контент

Вкладки:

* посты;
* фото;
* видео;
* комментарии — можно не показывать на MVP.

### 8. Команды

Команды, в которых состоит пользователь.

## Новые пользователи

У новых пользователей может не быть истории.

Нельзя делать так, чтобы профиль выглядел «пустым и бесполезным».

Нужно показывать:

* возможность заполнить опыт по словам пользователя;
* подсказку, как подать заявку;
* рекомендации для первого мероприятия;
* блок «Пока нет подтверждённых участий».

---

# 8. 👥 Профиль команды

Профиль команды — лицо команды на платформе.

Он должен отвечать на вопрос:

> Можно ли этим людям доверять?

## Главные задачи

* показать, кто команда;
* показать опыт;
* показать подход к безопасности;
* показать мероприятия;
* привлечь участников;
* дать новичкам ориентир.

## Структура профиля команды

### 1. Верхний блок

Содержит:

* логотип;
* название;
* регион;
* статус;
* краткое описание;
* количество проведённых мероприятий;
* количество объектов;
* количество участников через платформу.

### 2. О команде

Описание:

* сколько лет в теме;
* какие форматы проводит команда;
* подход к безопасности;
* философия;
* для кого мероприятия;
* как проходит первый прыжок.

### 3. Мероприятия

Вкладки:

* ближайшие;
* прошедшие.

Карточка:

* фото;
* название;
* дата;
* объект;
* статус;
* свободные места.

### 4. Объекты

Список объектов, где команда проводила мероприятия:

* название;
* высота;
* тип;
* фото;
* количество событий.

### 5. Участники команды

Показывать:

* аватар;
* имя;
* роль;
* ссылку на профиль.

### 6. Медиа

Фото и видео команды:

* с мероприятий;
* закулисье;
* подготовка;
* отчёты.

### 7. Активность

Посты команды:

* анонсы;
* отчёты;
* новости;
* истории.

## Статус «Проверенная команда»

Статус выдаётся только вручную админом.

Он даёт:

* визуальное выделение;
* больше доверия;
* приоритет в списках — опционально;
* понятный сигнал новичкам.

Важно: статус не должен выдаваться автоматически по количеству мероприятий.

---

# 9. 🏗 Объекты

Объект — это место, связанное с историей мероприятий, команд и участников.

Страница объекта должна отвечать на вопрос:

> Что это за место и почему оно важно для сообщества?

## Главный принцип

На платформе нельзя публиковать опасную информацию.

Не публикуются:

* точные координаты;
* способы доступа;
* технические схемы;
* точки крепления;
* инструкции по самостоятельной организации прыжка;
* детали, которые могут подтолкнуть человека к опасным действиям.

## Карточка объекта

Содержит:

* название;
* тип: мост, здание, труба, скала, другое;
* высоту;
* регион;
* фото;
* короткое описание;
* количество мероприятий;
* команды, которые проводили там события.

## Страница объекта

### 1. Hero-блок

* большое фото;
* название;
* тип;
* высота;
* регион.

### 2. Базовая информация

* тип;
* высота;
* регион;
* статус: активный / редко используется / архивный.

Статус можно добавить позже.

### 3. Описание

Только общее описание:

* атмосфера;
* визуальный характер;
* история;
* почему объект интересен.

Без технических деталей.

### 4. Мероприятия

* ближайшие;
* прошедшие;
* команда;
* дата;
* статус.

### 5. Команды

Команды, которые проводили мероприятия на объекте.

### 6. Медиа

Фото и видео объекта.

### 7. История

Можно показывать:

* количество проведённых мероприятий;
* первый зафиксированный ивент;
* количество подтверждённых участников.

## Публичные и скрытые объекты

Нужно предусмотреть два режима:

* **Публичный объект** — виден в каталоге;
* **Скрытый объект** — виден только через мероприятие или только участникам.

Это нужно для безопасности и приватности.

# 10. 🪂 Система бейджей

Бейджи — это история и опыт, а не рейтинг.

Главный принцип:

> Бейджи не должны превращать сообщество в соревнование «кто круче».

## Принципы

* Никакого ручного ввода количества прыжков на MVP.
* Всё считается через подтверждённые мероприятия.
* Бейджи выдаются за участие, объекты, высоту, активность и роли.
* Часть бейджей автоматическая.
* Особые бейджи выдаются вручную.
* Бейджи не должны блокировать участие.

## 10.1 Бейджи за участие

Выдаются автоматически по количеству подтверждённых мероприятий:

* 1 мероприятие — «Первый прыжок»;
* 2 мероприятия — «Вернулся снова»;
* 5 мероприятий — «В теме»;
* 10 мероприятий — «Свой человек»;
* 25 мероприятий — «Постоянный участник»;
* 50 мероприятий — «Живёт этим»;
* 100 мероприятий — «Ядро сообщества».

## 10.2 Бейджи за объекты

За количество уникальных объектов:

* 1 объект — «Новый горизонт»;
* 3 объекта — «Исследователь»;
* 5 объектов — «Открывает новые места»;
* 10 объектов — «Путешественник»;
* 20 объектов — «Коллекционер объектов».

## 10.3 Бейдж конкретного объекта

Каждый объект может давать отдельный бейдж:

* «Северный мост»;
* «Башня X»;
* «Каньон Y».

Так формируется карта личного опыта.

## 10.4 Бейджи за высоту

Считаются по максимальной высоте объекта, где пользователь подтвердил участие:

* 30+ м — «Первый уровень»;
* 50+ м — «Выше страха»;
* 100+ м — «Сотка»;
* 150+ м — «Граница»;
* 200+ м — «Высота»;
* 300+ м — «Свободное падение».

## 10.5 Бейджи событий

Выдаются за участие в особых мероприятиях:

* фестиваль;
* Winter Jump;
* Night Jump;
* юбилейное событие;
* командный выезд.

Мероприятие должно быть заранее помечено организатором или админом как особое.

## 10.6 Ролевые бейджи

Для организаторов:

* создал 1 мероприятие — «Организатор»;
* 5 мероприятий — «Делает движ»;
* 10 мероприятий — «Держит уровень»;
* 25 мероприятий — «Опора комьюнити».

Для команды и участников:

* «В команде»;
* «Актив команды»;
* «Фотограф»;
* «Инструктор»;
* «Old School»;
* «Легенда».

Часть таких бейджей лучше выдавать вручную.

## 10.7 Бейджи за контент

* первый пост;
* первое фото;
* 10 фото;
* популярный пост;
* хроникёр.

Для MVP можно оставить только:

* первый пост;
* первое фото.

## 10.8 Социальные бейджи

Можно добавить позже:

* первый комментарий;
* активный участник;
* помогает новичкам.

## Логика начисления

После мероприятия:

1. Организатор нажимает «Мероприятие состоялось».
2. Система показывает список принятых участников.
3. Организатор отмечает фактических участников.
4. Система создаёт записи участия.
5. Система добавляет объект пользователю.
6. Система проверяет условия бейджей.
7. Система выдаёт новые бейджи.
8. Пользователь получает уведомление.

---

# 11. 📋 Требования к участникам

Требования отображаются в мероприятии как информационный блок.

Пример:

* опыт: 20+ прыжков;
* высота: от 100 м;
* новичкам: нет;
* возраст: 18+;
* физическая форма: без противопоказаний.

Важно:

* требования не блокируют подачу заявки автоматически;
* пользователь может подать заявку даже без истории на платформе;
* решение принимает организатор;
* организатор видит опыт пользователя, комментарий и профиль.

# 14. 🛡 Модерация

Модерация нужна, чтобы защищать пользователей и сохранять доверие.

Главный принцип:

> Минимальная, но достаточная модерация.

## Что нужно модерировать

* посты;
* комментарии;
* медиа;
* мероприятия;
* объекты;
* команды;
* пользователей;
* жалобы.

## Жалобы

Пользователь может пожаловаться на:

* пост;
* комментарий;
* пользователя;
* команду;
* мероприятие;
* объект.

Причины жалобы:

* опасная инструкция;
* спам;
* оскорбления;
* фейковая команда;
* незаконный контент;
* публикация координат или технических деталей;
* другое.

## Действия модератора

* оставить без изменений;
* скрыть контент;
* удалить контент;
* предупредить пользователя;
* временно заблокировать;
* заблокировать навсегда;
* скрыть команду;
* убрать статус проверенной команды;
* скрыть объект.

## Особо опасный контент

Нужно сразу удалять или скрывать:

* инструкции «как прыгнуть самому»;
* точки доступа;
* координаты;
* схемы крепления;
* технические инструкции;
* призывы к самостоятельным опасным действиям;
* контент, который может спровоцировать новичков на риск.

## Админка

Разделы:

* жалобы;
* пользователи;
* команды;
* контент;
* объекты;
* мероприятия;
* бейджи;
* системные настройки.

Для MVP достаточно:

* список жалоб;
* удаление/скрытие постов и комментариев;
* блокировка пользователей;
* выдача статуса «проверенная команда»;
* скрытие объектов и мероприятий.

# 16. 🧭 Принципы продукта

## 1. Минимум нагрузки на организаторов

Организатор делает только главное:

* создаёт мероприятие;
* управляет заявками;
* после события подтверждает фактических участников.

## 2. Нет жёстких автоматических ограничений

Система помогает, но не принимает решение за человека.

## 3. Упор на реальный опыт

История строится вокруг:

* событий;
* объектов;
* высоты;
* команд;
* подтверждённого участия.

## 4. Сообщество важнее функций

Главное — люди, атмосфера, доверие, события и истории.

## 5. Безопасность важнее вирусности

Нельзя гнаться за контентом, который провоцирует опасные действия.

## 6. Бейджи — не рейтинг

Бейджи показывают путь человека, а не создают соревнование.

# 17. 🧱 Что важно не забыть в архитектуре продукта

## 17.1 Журнал действий

Нужен лог важных действий:

* кто принял заявку;
* кто отклонил заявку;
* кто подтвердил участие;
* кто изменил мероприятие;
* кто выдал ручной бейдж;
* кто скрыл объект;
* кто удалил контент.

Это поможет разбирать спорные ситуации.

## 17.2 Черновики

Нужны черновики:

* мероприятий;
* постов;
* возможно страниц объектов.

## 17.3 История изменений мероприятия

Минимально нужно хранить:

* перенос даты;
* отмену;
* изменение лимита мест;
* закрытие набора.

## 17.4 Резервный список

Можно добавить после MVP.

Сценарий:

* мест нет;
* пользователь всё равно оставляет заявку в резерв;
* организатор может принять его позже.

## 17.5 Ограничение новых аккаунтов

Чтобы уменьшить спам:

* лимит постов для новых пользователей;
* жалобы;
* временное ограничение комментирования при нарушениях.

# 18. 🧪 Проверка концепции перед разработкой

Перед активной разработкой стоит проверить несколько гипотез:

## Гипотеза 1

Команды готовы создавать мероприятия на платформе, если форма занимает 1–2 минуты.

## Гипотеза 2

Участники хотят иметь профиль с историей объектов, мероприятий и высоты.

## Гипотеза 3

Организаторам удобно принимать заявки внутри сайта, а не только через мессенджеры.

## Гипотеза 4

Лента оживёт, если автоматически создавать анонсы мероприятий и стимулировать отчёты после событий.

## Гипотеза 5

Новичкам нужен отдельный безопасный вход: первый прыжок, выбор команды, FAQ, базовые правила.

# 21. 🧰 Рекомендуемый технологический стек

## Основной стек для MVP

* **Next.js** — фронтенд и backend-for-frontend в одном проекте;
* **TypeScript** — типизация всей кодовой базы;
* **PostgreSQL** — основная база данных;
* **Prisma** — ORM для быстрого старта и удобных миграций;
* **Supabase Auth** — регистрация, вход, сессии;
* **Yandex Object Storage** — хранение фото и видео на старте;
* **Tailwind CSS** — стили;
* **shadcn/ui** — готовые UI-компоненты;
* **Zod** — валидация форм и API;
* **React Hook Form** — формы;
* **UploadThing или Supabase Storage SDK** — загрузка файлов;
* **Resend** — email-уведомления;
* **Vercel** — деплой фронтенда и серверной части;
* **Sentry** — ошибки и мониторинг;
* **Plausible / Umami** — простая аналитика.

## Почему такой стек

Этот стек хорошо подходит для MVP, потому что позволяет быстро собрать:

* страницы;
* формы;
* авторизацию;
* роли;
* базу данных;
* загрузку медиа;
* админку;
* уведомления;
* SEO-страницы мероприятий, команд и объектов.

## Альтернатива ORM

Можно выбрать **Drizzle** вместо Prisma, если хочется:

* больше контроля над SQL;
* более лёгкий слой ORM;
* ближе работать с PostgreSQL.

Для быстрого MVP проще Prisma. Для более «ручной» архитектуры и сильного контроля над запросами — Drizzle.

## Хранение медиа на вырост

На старте достаточно Supabase Storage.

Когда фото и видео станет много, можно вынести медиа в:

* Cloudflare R2;
* S3-compatible storage;
* отдельный CDN.

# 24. ✅ Самый правильный первый MVP

Самая первая рабочая версия должна быть не «соцсетью про роупджампинг», а сервисом мероприятий и заявок.

Минимальное ядро:

1. Пользователь регистрируется.
2. Команда создаёт страницу.
3. Организатор создаёт мероприятие.
4. Пользователь подаёт заявку.
5. Организатор принимает или отклоняет заявку.
6. После события организатор подтверждает фактических участников.
7. Участие появляется в профиле.
8. Пользователь получает базовые бейджи.
9. Мероприятие остаётся в истории команды и объекта.

Это ядро можно развивать дальше без переделки всей архитектуры.

# 25. Главный вывод

Проект стоит строить вокруг реального действия:

> человек нашёл мероприятие → подал заявку → попал на прыжок → получил историю → вернулся снова

Лента, посты, бейджи, объекты и профили должны усиливать этот сценарий, а не заменять его.

Главная ценность платформы — не в количестве функций, а в том, что она соединяет реальные команды, реальные мероприятия и реальных людей.

Как масштабировать T3-проект правильно
1. Начать как модульный монолит

На старте не нужны микросервисы. Делай один Next.js-проект, но разделяй домены:

/features/events
/features/applications
/features/teams
/features/profiles
/features/objects
/features/feed
/features/badges
/features/moderation

То есть физически проект один, но логически он уже разделён.

Это позволит потом без боли вынести, например, медиа, уведомления или поиск в отдельный сервис.

2. Не завязываться на tRPC везде

tRPC хорош для внутреннего web-приложения: он даёт end-to-end type safety без генерации схем и хорошо сочетается с Next.js.

Но для будущего я бы сделал так:

Внутренний web-интерфейс → tRPC
Публичные API / mobile app / webhooks → REST route handlers

Например:

trpc.events.create
trpc.applications.accept
trpc.feed.createPost

А позже:

/api/public/events
/api/webhooks/storage
/api/mobile/events

Так ты не упрёшься в tRPC, если потом появится мобильное приложение.

3. Базу сразу проектировать нормально

Главное ограничение при росте будет не create-t3-app, а PostgreSQL и модель данных.

Сразу заложи:

users
profiles
teams
team_members
events
event_applications
event_participations
objects
posts
comments
media
badges
user_badges
reports
notifications
audit_logs

И очень важно: не хранить всё “JSON-ом в одной таблице”. Для твоего проекта нужны нормальные связи.

4. Object Storage — правильно

Твой Yandex Object Storage отлично подходит.

Лучше сразу сделать так:

Next.js backend генерирует signed URL
↓
frontend грузит файл напрямую в Yandex Object Storage
↓
в PostgreSQL сохраняется media record

Не надо гонять фото/видео через сервер приложения. Yandex Object Storage совместим с S3 API, поэтому можно использовать стандартный AWS S3 SDK.

5. Docker — хороший путь для будущего

Next.js можно деплоить не только на Vercel, но и как Node.js server или Docker container. Это официально поддерживаемые варианты деплоя.

С твоим Docker локально можно идти так:

локально:
Next.js app
PostgreSQL
Redis позже
MinIO опционально

prod:
Next.js Docker container
PostgreSQL managed/self-hosted
Yandex Object Storage
Nginx / Caddy
Где могут быть проблемы при масштабировании
1. Serverless + Prisma

Если деплоить на serverless-платформы, нужно следить за количеством подключений к базе. У Prisma есть решения вроде connection pooling / Accelerate для таких сценариев, потому что при росте нагрузки соединения с БД могут стать узким местом.

Если будешь деплоить в Docker как обычный Node.js server — будет проще контролировать подключения.

2. Медиа

Фото и видео быстро станут тяжёлой частью проекта.

Сразу делай:

media table:
id
ownerId
bucket
key
type
mimeType
size
width
height
createdAt

А не просто imageUrl строкой в посте.

3. Лента

Лента может стать самым тяжёлым местом.

На MVP можно просто:

posts order by createdAt desc

Позже добавить:

индексы
пагинацию cursor-based
кэширование
ленты по региону
ленты по подпискам
4. Уведомления

На старте можно писать уведомления прямо в БД.

Позже вынести в очередь:

event completed
→ recalculate badges
→ create notifications
→ send email

Тут пригодится Redis + BullMQ или аналог.
