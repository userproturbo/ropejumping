Конспект проекта ropejumping
1. Идея проекта
Проект — это платформа для русскоязычного ropejumping-сообщества.
Основная цель:
дать командам, организаторам и участникам единое место,где можно создавать команды, публиковать мероприятия,принимать заявки, вести историю участий, показывать бейджи,объекты, ленту и модерировать контент.
Сейчас проект движется в правильной логике: сначала делаем продуктовую механику и безопасность, потом дизайн.

Что уже сделано
2. Базовая инфраструктура
Сделано:
- Next.js приложение;- Prisma + PostgreSQL;- Auth.js авторизация;- tRPC API;- базовая структура router-ов;- русскоязычный интерфейс;- базовая навигация;- адаптивная шапка;- README постепенно обновляется под каждую фичу.
Проект уже имеет рабочую основу, на которую можно наращивать полноценный продукт.

3. Авторизация и профили
Сделано:
- пользователь может войти в аккаунт;- у пользователя может быть публичный профиль;- профиль содержит username, отображаемое имя, город, аватар, bio, внешний опыт;- публичный профиль доступен по /u/[username];- многие важные действия требуют заполненный профиль.
Профиль сейчас нужен для:
- подачи заявки на мероприятие;- создания команды;- создания объекта;- публикации поста;- отправки жалобы;- подачи заявки в команду.
Это правильно, потому что платформа завязана на доверие между людьми.

4. Команды
Сделано:
- публичный список команд;- публичная страница команды;- создание команды;- редактирование команды;- логотип команды;- настройки команды;- мои команды;- роли доступа в команде.
Роли доступа:
OWNER       владелецADMIN       администраторORGANIZER   организаторMEMBER      участник
Важное решение:
OWNER / ADMIN / ORGANIZER / MEMBER — это роли доступа,а не профессии или функции человека.
Это значит, что только эти роли влияют на права.

5. Управление участниками команды
Сделано:
- OWNER/ADMIN могут добавлять участников по username;- OWNER/ADMIN могут менять роль доступа участника;- OWNER/ADMIN могут удалять участников;- OWNER защищён от удаления и смены access-роли;- появилась отдельная страница управления участниками.
Также добавлены функциональные роли:
OPERATOR        ОператорPHOTOGRAPHER    ФотографMEDIC           МедикINSTRUCTOR      ИнструкторCOORDINATOR     КоординаторRADIO_OPERATOR  Связист
Важное решение:
Функциональные роли не дают прав доступа.
То есть человек может быть, например:
MEMBER + Фотограф + Связист
Но это не делает его администратором или организатором.

6. Заявки на вступление в команду
Сделано:
- пользователь с профилем может подать заявку в публичную команду;- пользователь может написать сообщение к заявке;- пользователь может отменить свою pending-заявку;- OWNER/ADMIN могут смотреть заявки;- OWNER/ADMIN могут принять или отклонить заявку;- при принятии заявки пользователь становится MEMBER;- email пользователей не раскрывается;- история заявок сохраняется.
Важное решение:
Принятая заявка всегда создаёт TeamMember с ролью MEMBER.Повышение до ORGANIZER/ADMIN делается отдельно вручную.
Это безопасно.

Мероприятия сейчас являются центральной сущностью проекта.

Статусы:

10. Завершение мероприятия
Сделано:

Это уже даёт платформе историю участия.

Важные продуктовые решения
17. Дизайн пока не финализируем
Правильное решение:
сначала доделать ключевую логику,потом делать цельный дизайн всего сайта.
Почему:
- макет может измениться;- структура страниц ещё расширяется;- появляются новые сущности;- рано делать pixel-perfect UI, пока меняется продуктовая логика.
Сейчас можно делать только минимальную удобность: чтобы тестировать было комфортно.

18. Безопасность контента
У проекта есть чувствительная тема, поэтому уже заложены важные ограничения:
- не публиковать точные координаты;- не публиковать способы доступа;- не публиковать точки крепления;- не публиковать технические схемы;- объекты создают только команды;- заявки требуют профиль;- модерация уже есть.

19. Task 018: состав мероприятия
Это самый логичный следующий большой шаг.
Сейчас есть:
- участники команды;- функциональные роли в команде;- заявки на мероприятие;- подтверждённые участники.
Но пока нет отдельного состава конкретного мероприятия.
Нужно добавить сущность вроде:
EventCrewMember
Она будет отвечать за то, кто работает на конкретном мероприятии.
Например:
Мероприятие: Климатический мостСостав:- Иван — Инструктор- Олег — Фотограф- Анна — Медик- Сергей — Связист- Дмитрий — Координатор
Важно:
функции человека в команде и функции на конкретном мероприятии могут отличаться.
Например, человек в команде может быть фотографом, но на конкретном мероприятии быть координатором.
Что сделать:
- добавить EventCrewMember;- привязать к Event;- выбирать людей из TeamMember;- назначать функции на мероприятие;- показывать состав на странице мероприятия;- показывать состав в карточке мероприятия, если компактно возможно;- не смешивать crew с заявками участников.

Текущее состояние проекта коротко
Сейчас проект уже умеет:
- регистрировать пользователей;- создавать профили;- создавать команды;- управлять участниками команд;- принимать заявки в команды;- создавать объекты от имени команды;- создавать мероприятия;- управлять статусами мероприятий;- принимать заявки на мероприятия;- завершать мероприятия;- подтверждать участников;- начислять бейджи;- публиковать посты;- загружать изображения;- модерировать жалобы;- работать на русском языке.
Это уже не просто заготовка, а рабочий MVP-скелет платформы.

Главный следующий продуктовый шаг
Я бы следующим делал:
Task 018: Состав мероприятия
Потому что твоя идея очень правильная:
организатор создаёт мероприятие,добавляет туда людей из команды,назначает им функции,а пользователи сразу видят, кто проводит мероприятие.
Это сильно повышает доверие к мероприятию.
Плюс туда же можно аккуратно добавить:
- принятых участников;- подтверждённых участников после завершения;- разные блоки на странице мероприятия.
И это будет логичным продолжением всех уже сделанных командных ролей 👥
